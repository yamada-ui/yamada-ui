You are a Yamada UI contributor. Your task is to fix a GitHub issue end-to-end: analyze it, implement the fix, write tests, and submit a PR — all following yamada-ui conventions defined in CLAUDE.md.

## Language Selection

**Before doing anything else**, ask the user which language they want for communication:

- 🇯🇵 日本語 (Japanese)
- 🇬🇧 English

Use `AskUserQuestion` to ask this. Then use the selected language for **all** your own prose: status updates, summaries, questions, and explanations. Do not translate tool output, error messages, file paths, code, or command output. Code, commit messages, changeset descriptions, and PR content remain in **English** regardless of language choice.

## Input

The issue reference is: $ARGUMENTS

Parse the input:

- Full URL (e.g., `https://github.com/yamada-ui/yamada-ui/issues/123`) → extract the numeric issue number
- `#123` or bare `123` → strip `#` and use as-is

Validate that the resolved value is a positive integer. If `$ARGUMENTS` is empty or not a valid issue reference, ask the user to provide the issue number or URL before proceeding.

The repository is always `yamada-ui/yamada-ui`.

---

## Step 1: Fetch & Analyze the Issue

Run `gh issue view <number> --repo yamada-ui/yamada-ui` to get the issue details.

Classify the issue:

- **bug**: Something is broken or behaving incorrectly
- **feat**: New feature or enhancement
- **docs**: Documentation change
- **test**: Test improvement
- **refactor**: Code restructuring without behavior change
- **chore**: Maintenance, dependency updates, CI/CD changes

Identify the affected package and component/area:

- `@yamada-ui/react` → `packages/react/src/components/<name>/` or `packages/react/src/hooks/<name>/`
- `@yamada-ui/utils` → `packages/utils/src/`
- `@yamada-ui/cli` → `packages/cli/src/`
- `www` (docs site) → `www/contents/`

**Summarize your analysis to the user before proceeding.** If the issue is unclear or lacks enough information to act on, stop and explain what's missing.

---

## Step 2: Create a Branch

**First, check for uncommitted changes:**

```bash
git status --short
```

If there are uncommitted changes, alert the user and do not proceed until they confirm how to handle them (stash, commit, or discard).

**Then, check for branch name conflicts:**

```bash
git switch main && git pull --ff-only origin main
git branch --list "<type>/<scope>"
```

If a branch with the target name already exists, ask the user to choose a different name or confirm deletion of the existing branch.

```bash
git switch -c <type>/<scope>
```

Branch naming by type:

- Bug fix → `fix/<component-or-area>`
- Feature → `feat/<component-or-area>`
- Docs → `docs/<component-or-area>`
- Test → `test/<component-or-area>`
- Refactor → `refactor/<component-or-area>`

---

## Step 3: Explore & Identify Files

Map the issue to file locations. Read the relevant source files, tests, and stories before making any changes. Understand the existing code thoroughly.

Key patterns to look for:

- Single-part components use `createComponent` / `withContext`
- Multi-part compound components use `createSlotComponent` / `withContext` (e.g., accordion, tabs, drawer, card)
- Component files almost always have `"use client"` directive — omit only for pure render utilities that use no hooks or browser APIs (e.g., `<Show>`, `<For>`)
- Class names are automatically generated by `createComponent`/`createSlotComponent` as `ui-<name>` or `ui-<name>__<slot>` — do not set class names manually
- Test files use `import { a11y, render, screen } from "#test"`

---

## Step 4: Implement the Fix

Make the minimal changes needed to address the issue. Follow the existing code patterns and conventions.

Critical rules from CLAUDE.md:

- **Do not bundle multiple fixes.** Only fix what the issue describes.
- **Accessibility is required.** All components must support ARIA attributes, keyboard navigation, and screen readers.
- **Never manually edit generated files** like `props.json` or `registry.json`. Generated files typically have an auto-generated header comment. Use the generation commands (`pnpm gen:tokens`, `pnpm gen:styles`, `pnpm gen:icons`, `pnpm gen:props`) if regeneration is needed.

---

## Step 5: Write or Update Tests

Tests are **required** for bug fixes and new features.

- **Bug fix**: Write a test that would have caught the bug.
- **Feature**: Test new behavior, edge cases, and accessibility.
- **Docs-only changes**: No tests needed.

Test conventions:

```typescript
import { a11y, render, screen } from "#test"
```

Run targeted tests:

```bash
# For components:
pnpm react test:jsdom --run src/components/<name>/
# For hooks:
pnpm react test:jsdom --run src/hooks/<name>/
```

---

## Step 6: Run Quality Checks

Run the quality checks for **all** affected packages:

```bash
pnpm react quality   # For @yamada-ui/react
pnpm utils quality   # For @yamada-ui/utils
pnpm cli quality     # For @yamada-ui/cli
pnpm www quality     # For www
```

If your changes touch multiple packages (e.g., both `packages/react/` and `www/`), run quality checks for each affected package.

**Fix all failures before proceeding.** Do not skip this step.

---

## Step 7: Create a Changeset (if needed)

A changeset is **required** when modifying `packages/{cli,react,utils}/src/**/*.{ts,tsx}` (excluding test and story files).

**Skip** for docs-only, test-only, or CI/config changes.

Create a file in `.changeset/` with a randomly-generated name consistent with changesets convention (e.g., `purple-foxes-smile.md`). Check that no file with the same name already exists.

```md
---
"@yamada-ui/react": patch
---

One-sentence summary of the fix in English.
```

If your changes span multiple packages, list all affected packages:

```md
---
"@yamada-ui/react": patch
"@yamada-ui/utils": patch
---

One-sentence summary of the fix in English.
```

Bump types:

- `patch` — bug fix or internal change
- `minor` — new feature (backward compatible)
- `major` — breaking change

---

## Step 8: Commit

Run `git status --short` to review all changes. Verify no unexpected files appear.

Stage only the specific files you changed (never use `git add .` or `git add -A`):

```bash
git add <specific-files>
```

Commit with a conventional commit message:

```
<type>(<scope>): <description>
```

Lefthook will automatically run ESLint, Prettier, and commitlint. If the hooks fail, fix the issues and create a **new** commit (do not amend).

---

## Step 9: Push & Create PR

**Before pushing, ask the user for confirmation.** Show them:

- The branch name
- A summary of all changes (`git diff main..HEAD --stat`)
- The commit message

Only proceed if the user confirms.

Push the branch:

```bash
git push -u origin <branch-name>
```

Create the PR using the exact template structure from `.github/pull_request_template.md`. Evaluate whether the fix is a breaking change — if yes, describe the impact and migration path.

```bash
gh pr create --repo yamada-ui/yamada-ui --base main --title "<type>(<scope>): <description>" --body "$(cat <<'EOF'
Closes #<issue-number>

## AI used

- [ ] I did not use AI to create this PR.
- [x] (If there is no check above) I checked the generated content before submitting.

## Description

<Brief description of what was changed and why.>

## Current behavior (updates)

<Describe the current behavior being modified, or "N/A" for new features.>

## New behavior

<Describe the new behavior or changes this PR adds.>

## Is this a breaking change (Yes/No):

<Evaluate and answer Yes or No. If Yes, describe the impact and migration path.>

## Additional Information
EOF
)"
```

Share the PR URL with the user when done.

After creating the PR, check the CI status:

```bash
gh run list --repo yamada-ui/yamada-ui --branch <branch-name> --limit 1
```

If CI fails, inform the user and offer to fix the issues.

---

## Edge Cases

- **Issue is unclear**: Stop after Step 1 and explain what information is missing. Do not guess.
- **Multiple unrelated areas affected**: Fix only what the issue describes. If you notice other issues, mention them but do not fix them.
- **Generated files need updating**: Run the appropriate generation command (e.g., `pnpm gen:props`) rather than editing generated files manually.
- **Quality checks fail**: Fix all failures before committing. Do not skip quality checks.
- **Branch already exists**: Ask the user before reusing or deleting an existing branch.
- **Uncommitted changes**: Never silently switch branches with dirty working tree. Alert the user first.
